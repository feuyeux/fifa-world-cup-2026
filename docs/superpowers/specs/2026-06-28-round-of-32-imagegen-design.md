# 1/16 决赛 16 张海报直出方案

## 目标

基于仓库中的 `x.json`、`squads_json/*.json` 与 `group_kits/*.png`，直接使用 `imagegen` 生成 16 张 2026 世界杯 1/16 决赛对阵海报，并保存到 `16/` 目录。

## 已确认约束

- 不使用脚本驱动批量生成。
- 直接使用 `imagegen` 逐张生成。
- 每张图展示双方完整 26 人名单。
- 阵容必须按 `GK / DF / MF / FW` 分块。
- 每行最多 7 人，超过必须换行。
- 输出为高清 PNG。

## 输入来源

### 赛程与全局风格

`x.json` 作为赛程骨架与全局版式风格来源，提供：

- 赛事名称、阶段、日期、北京时间。
- 场馆、城市、海拔、天气。
- 统一的视觉风格约束：
  - 横版比例。
  - 浅暖纸张纹理背景。
  - 彩色钢笔画体育商业信息图风格。
  - 中间明显 `VS` 视觉符号。
  - 对称、留白、清晰可读。

### 阵容数据

若 `x.json` 某场比赛的 `roster` 不完整或为 `TODO`，则从对应的 `squads_json/*.json` 补齐完整 26 人名单。

补齐规则：

- 按球队英文名匹配到对应 JSON 文件。
- 从 `players` 中提取 26 人完整名单。
- 按 `GK -> DF -> MF -> FW` 排序。
- 每个位置内按号码升序。

### 球衣信息

球衣视觉参考使用 `group_kits/<group>.png`。

使用规则：

- 默认外场球员使用主场球衣风格。
- 门将必须使用与外场明显不同的守门员球衣风格。
- prompt 中只提炼关键球衣信息，不逐字穷举图中细节，避免造成生成不稳定。
- 若同组图不足以明确队伍归属，可辅以 `team_jerseys/*.png` 做队伍校正，但主参考仍为 `group_kits/`。

## 输出设计

### 画面结构

每张海报采用统一结构：

- 顶部：
  - 左侧比赛时间与日期。
  - 中间主标题与双方队名。
  - 右侧球衣说明。
- 中部：
  - 左右两队完整阵容对称排布。
  - 中间放置醒目的 `VS` 视觉元素。
- 底部：
  - 不使用页脚。

### 阵容展示规则

- 每队按 `GK / DF / MF / FW` 分块显示。
- 每名球员使用半身像。
- 所有半身像等高。
- 不裁切头部或肩部。
- 不重叠、不挤压。
- 每行最多显示 7 名球员。
- 当某一位置人数超过 7 名时自动换行。
- 球员之间保留独立空间。
- 位置块之间保持清晰分隔。

### 文案规则

- 中英双语信息可读。
- 英文名使用大写姓氏或仓库现有统一风格。
- 所有文字必须锐利、不可重叠、不可遮挡球员。
- 重点信息优先级：
  - 对阵双方。
  - 比赛时间。
  - 场馆与城市。
  - 天气与海拔。

## 文件输出

输出目录：`16/`

命名规则：

- `16/knockout_01_south-africa_vs_canada.png`
- `16/knockout_02_brazil_vs_japan.png`
- ...
- `16/knockout_16_colombia_vs_ghana.png`

命名与仓库当前 `knockout_*.png` 风格保持一致，但新产物集中保存到 `16/`。

## 执行策略

采用逐场执行，而非一次性并发生成。

原因：

- 便于控制每张图的 prompt 稳定性。
- 若某一场失败，只需重试该场。
- 便于即时校验是否成功保存到目标目录。

执行顺序：

1. 先生成第 1 场并验证能否正常写入 `16/`。
2. 验证通过后，按 1 到 16 顺序继续生成剩余海报。
3. 每张图生成后立即确认文件名与保存位置。

## 校验标准

最终完成条件：

- `16/` 目录下存在 16 张 PNG。
- 每张图都对应正确场次与双方球队。
- 每张图都展示双方完整 26 人名单。
- 每张图都遵守 `GK / DF / MF / FW` 分块。
- 每行最多 7 人。
- 门将穿着明显区别于外场球员。
- 整体风格与 `x.json` 的全局设定一致。

## 风险与处理

### 风险 1：球衣细节不稳定

处理：

- 保持 prompt 中球衣描述简洁，只强调主色、主场属性和门将差异。

### 风险 2：26 人双边排版过密

处理：

- 强调横版、高度增加、宽留白、位置块换行和 generous spacing。

### 风险 3：个别场次输出失败或未写入

处理：

- 逐张检查，仅重试失败场次。

## 不在本次范围内

- 不修改仓库中的现有 `knockout_*.png`。
- 不重写 `x.json` 或批量生成脚本。
- 不切换为首发 11 人海报方案。
